## Priority 1: Critical (Must Fix)

### Issue #1: Unbounded Memory Allocation

**Location**: `packages/core/src/client.ts:24`

**Severity**: Critical
**Category**: Security / Performance

**Description**:
The message queue has no size limit. A malicious client or network issue could cause unbounded memory growth.

```typescript
protected messageQueue: string[] = []
```

**Why it matters**:
- DoS vulnerability: A malicious client can send messages while disconnected to exhaust server memory
- Production stability: Network issues could cause legitimate memory spikes
- No recovery mechanism once queue is full

**Recommendation**:
Implement a maximum queue size with eviction policy:

```typescript
private readonly MAX_QUEUE_SIZE = 1000

send(message: IMessage) {
  const payload = JSON.stringify(message)
  if (this.status === 'open' && this.socket?.readyState === WebSocket.OPEN) {
    this.socket.send(payload)
  } else {
    if (this.messageQueue.length >= this.MAX_QUEUE_SIZE) {
      this.messageQueue.shift() // Drop oldest message
      console.warn('Message queue full, dropping oldest message')
    }
    this.messageQueue.push(payload)
  }
}
```

### Issue #3: Race Condition in Client Removal

**Location**: `packages/core/src/services/socket.ts:108-115`

**Severity**: Critical
**Category**: Correctness

**Description**:
The socket close handler checks if the socket matches, but between the close event firing and the check, the realm could have been modified.

```typescript
socket.on('close', () => {
  // Race condition check: Ensure we are removing the specific client instance
  // that owns this socket.
  if (newClient.getSocket() === socket) {
    this.realm.removeClientById(newClient.getId())
    this.emit('close', newClient)
  }
})
```

**Why it matters**:
- If a client reconnects quickly with the same ID, the old close handler could remove the new client
- Could cause unexpected disconnections for active clients
- Timing-dependent bug that's difficult to reproduce

**Recommendation**:
Use a unique identifier for each connection instance instead of socket reference comparison:

```typescript
// Add a connection token to Client class
private readonly connectionToken: symbol

// In socket.ts, use the token for comparison
socket.on('close', () => {
  if (newClient.getConnectionToken() === connectionToken) {
    this.realm.removeClientById(newClient.getId())
    this.emit('close', newClient)
  }
})
```

### Issue #4: JSON Injection via Malformed Messages

**Location**: `packages/core/src/Dispatcher.ts:27-36`

**Severity**: High
**Category**: Security

**Description**:
Malicious JSON could cause prototype pollution or excessive memory usage through deeply nested objects.

```typescript
processRaw(data: string | Buffer, sender?: any) {
  try {
    const message = JSON.parse(data.toString())
    this.emit(message, sender)
    return message
  } catch (e) {
    console.error('Dispatcher: Failed to parse message', e)
    return null
  }
}
```

**Why it matters**:
- No message size limits
- No structure validation
- Potential for prototype pollution attacks
- Memory exhaustion through deeply nested objects

**Recommendation**:
Implement message size limits and schema validation:

```typescript
private readonly MAX_MESSAGE_SIZE = 1024 * 1024 // 1MB

processRaw(data: string | Buffer, sender?: any) {
  try {
    const dataStr = data.toString()

    // Check size before parsing
    if (dataStr.length > this.MAX_MESSAGE_SIZE) {
      console.error('Dispatcher: Message exceeds maximum size')
      return null
    }

    const message = JSON.parse(dataStr)

    // Validate message structure
    if (!message || typeof message !== 'object') {
      console.error('Dispatcher: Invalid message format')
      return null
    }

    this.emit(message, sender)
    return message
  } catch (e) {
    console.error('Dispatcher: Failed to parse message', e)
    return null
  }
}
```

### Issue #7: Inefficient Subscriber Iteration

**Location**: `packages/core/src/Synnel.ts:114-123`

**Severity**: Medium
**Category**: Performance

**Description**:
The relay handler iterates through all subscribers synchronously. The code acknowledges this as a TODO but has no mitigation.

```typescript
// Optimization: If you have 10,000 users, don't iterate one by one.
// For now (MVP), iteration is fine.
subscribers.forEach((id: string) => {
  if (id === senderId) return // Don't echo back to sender
  const targetClient = this.realm.getClientById(id)
  if (targetClient) {
    targetClient.send(message)
  }
})
```

**Why it matters**:
- At 10,000+ subscribers, this blocks the event loop
- Could cause message delivery delays
- Scales poorly for large applications

**Recommendation**:
Use batching to avoid event loop starvation:

```typescript
private async handleRelay(client: IClient, message: IMessage): Promise<void> {
  const channelName = message.channel
  if (!channelName) return

  const subscribers = Array.from(this.subscriptionManager.getSubscribers(channelName))
  const senderId = client.getId()

  // Batch sends to avoid event loop starvation
  const BATCH_SIZE = 100
  for (let i = 0; i < subscribers.length; i += BATCH_SIZE) {
    const batch = subscribers.slice(i, i + BATCH_SIZE)
    await Promise.all(
      batch
        .filter((id) => id !== senderId)
        .map((id) => {
          const targetClient = this.realm.getClientById(id)
          return targetClient?.send(message)
        })
    )
    // Allow event loop to process other tasks
    await new Promise(resolve => setImmediate(resolve))
  }
}
```

### Issue #10: No Rate Limiting

**Location**: `packages/core/src/services/socket.ts:117`

**Severity**: High
**Category**: Security

**Description**:
Clients can send unlimited messages, enabling DoS attacks against the server.

```typescript
socket.on('message', (data) => {
  try {
    const message = JSON.parse(data.toString())
    this.emit('message', newClient, message)
  } catch (e) {
    // ...
  }
})
```

**Why it matters**:
- Single client can exhaust server resources
- No protection against message flooding
- Could degrade service for all users

**Recommendation**:
Implement per-client rate limiting:

```typescript
private readonly clientMessageCounts = new Map<WebSocket, { count: number; resetTime: number }>()
private readonly RATE_LIMIT = 100 // messages per minute
private readonly RATE_WINDOW = 60000 // 1 minute

socket.on('message', (data) => {
  const now = Date.now()
  const state = this.clientMessageCounts.get(socket) || {
    count: 0,
    resetTime: now + this.RATE_WINDOW
  }

  if (now > state.resetTime) {
    state.count = 0
    state.resetTime = now + this.RATE_WINDOW
  }

  state.count++
  this.clientMessageCounts.set(socket, state)

  if (state.count > this.RATE_LIMIT) {
    socket.send(
      JSON.stringify({
        type: MessageType.ERROR,
        data: {
          message: 'Rate limit exceeded',
          code: 'RATE_LIMIT_EXCEEDED'
        },
      }),
    )
    socket.close(1008, 'Rate limit exceeded')
    return
  }

  // Process message...
})
```

### Issue #14: Reconnection Backoff Lacks Jitter

**Location**: `packages/core/src/ConnectionManager.ts:14-18`

**Severity**: Low
**Category**: Performance

**Description**:
Exponential backoff doesn't add random jitter, which can cause "thundering herd" problems when many clients disconnect simultaneously.

```typescript
nextDelay(attempt: number): number {
  return Math.min(
    this.baseDelay * Math.pow(this.factor, attempt),
    this.maxDelay,
  )
}
```

**Why it matters**:
- If server restarts, all clients reconnect at nearly the same time
- Could cause load spikes
- Best practice for distributed systems

**Recommendation**:
Add jitter to prevent synchronized reconnection:

```typescript
nextDelay(attempt: number): number {
  const baseDelay = Math.min(
    this.baseDelay * Math.pow(this.factor, attempt),
    this.maxDelay,
  )
  // Add jitter: ±25% random variation
  const jitter = baseDelay * 0.25 * (2 * Math.random() - 1)
  return Math.max(0, baseDelay + jitter)
}
```

### Issue #15: Channels Never Deleted

**Location**: `packages/core/src/models/subscribe.ts:94`

**Severity**: Medium
**Category**: Memory / Performance

**Description**:
Channels are never deleted even when empty. Could cause memory leak with dynamic channel names (e.g., user-specific channels).

```typescript
private unsubscribe(client: IClient, channelName: string): void {

    channel.unsubscribe(clientId)
    // Removed auto-delete to give server full control over channel lifecycle
  }
  // ...
```

**Why it matters**:
- Memory leak for applications with ephemeral channels
- No way to clean up unused channels
- Channel registry grows unbounded

**Recommendation**:
Make channel cleanup configurable:

```typescript
interface SubscriptionManagerOptions {
  autoDeleteEmptyChannels?: boolean
  emptyChannelGracePeriod?: number // ms before deletion
}

// In unsubscribe method:
if (channel.subscribers.size === 0 && this.options.autoDeleteEmptyChannels) {

      this.channels.delete(channelName)
```
